`dspy.Evaluate` is your benchmarking tool. Before you optimize, you run it to establish a baseline score. After you optimize, you run it again to measure how much your program improved. It executes your program on every example in a dataset, calls your metric on each result, and returns the average score as a percentage. Running evaluation before and after every optimization cycle gives you a clear, reproducible picture of progress and helps you catch regressions early. You can also use it to debug failures by inspecting per-example outputs.

## Constructor

```python
evaluate = dspy.Evaluate(
    devset,
    metric,
    num_threads=1,
    display_progress=True,
    display_table=0,
    return_outputs=False,
)
```

### Parameters

<ParamField path="devset" type="list[dspy.Example]" required>
  The evaluation dataset. Each example is passed to your program as an input, and then both the example and the prediction are passed to your metric.
</ParamField>

<ParamField path="metric" type="callable" required>
  A scoring function with signature `(example, prediction, trace) -> float | bool`. The scores are averaged across all examples to produce the final result.
</ParamField>

<ParamField path="num_threads" type="int" default="1">
  Number of parallel threads to use when running your program on the dataset. Increase this to speed up evaluation on large datasets — values between 4 and 16 are common depending on your provider's rate limits.
</ParamField>

<ParamField path="display_progress" type="bool" default="true">
  When `True`, a tqdm progress bar is shown during evaluation so you can track how many examples have been processed.
</ParamField>

<ParamField path="display_table" type="int" default="0">
  Number of example results to display in a formatted table after evaluation completes. Set to `5` or `10` for a quick sanity check, or `0` to suppress table output entirely.
</ParamField>

<ParamField path="return_outputs" type="bool" default="false">
  When `True`, calling the evaluator returns a tuple of `(score, outputs)` where `outputs` is a list of `(example, prediction, score)` triples — one per dataset example. Use this when you need to inspect individual predictions.
</ParamField>

---

## Calling the Evaluator

Calling the `evaluate` object with a program runs the full evaluation and returns the aggregate score:

```python
score = evaluate(program)
print(f'Score: {score:.1%}')  # e.g. Score: 74.0%
```

---

## Comparing Programs

The most common pattern is to evaluate your baseline and optimized programs back-to-back using the same `evaluate` instance:

```python
evaluate = dspy.Evaluate(devset=dev, metric=my_metric, num_threads=8)

baseline = evaluate(program)
optimized = evaluate(optimized_program)

print(f'Baseline:  {baseline:.1%}')
print(f'Optimized: {optimized:.1%}')
print(f'Improvement: +{(optimized - baseline):.1%}')
```

---

## Inspecting Per-Example Outputs

When you need to understand why your program is failing on specific examples, set `return_outputs=True` to get a full list of individual predictions alongside their scores:

```python
score, outputs = dspy.Evaluate(
    devset=dev,
    metric=my_metric,
    return_outputs=True,
)(program)

for example, prediction, score in outputs:
    if score == 0:
        print('Failed:', example.question)
        print('Predicted:', prediction.answer)
        print('Expected:', example.answer)
```

<Tip>
  Sort the `outputs` list by score to quickly surface your worst-performing examples and identify patterns in the failures. Those failure patterns are exactly what you should communicate to your metric's `feedback` field when using GEPA.
</Tip>

<Note>
  `dspy.Evaluate` is stateless — you can call the same instance on as many programs as you like. There is no need to re-instantiate it between baseline and optimized evaluations.
</Note>
