`dspy.Evaluate` is your benchmarking tool. Before you optimize, you run it to establish a baseline score. After you optimize, you run it again to measure how much your program improved. It executes your program on every example in a dataset, calls your metric on each result, and returns an `EvaluationResult` containing the aggregate score and the full list of per-example predictions. Running evaluation before and after every optimization cycle gives you a clear, reproducible picture of progress and helps you catch regressions early.

## Constructor

```python
evaluate = dspy.Evaluate(
    devset,
    metric,
    num_threads=None,
    display_progress=False,
    display_table=False,
)
```

### Parameters

<ParamField path="devset" type="list[dspy.Example]" required>
  The evaluation dataset. Each example is passed to your program as an input, and then both the example and the prediction are passed to your metric.
</ParamField>

<ParamField path="metric" type="callable" required>
  A scoring function with signature `(example, prediction, trace=None) -> float | bool`. Scores are aggregated across all examples: booleans become a pass-rate percentage, floats are averaged.
</ParamField>

<ParamField path="num_threads" type="int" default="None">
  Number of parallel threads to use when running your program on the dataset. Defaults to `None`, which runs sequentially. Increase this to speed up evaluation on large datasets — values between 4 and 16 are common depending on your provider's rate limits.
</ParamField>

<ParamField path="display_progress" type="bool" default="false">
  When `True`, a tqdm progress bar is shown during evaluation so you can track how many examples have been processed.
</ParamField>

<ParamField path="display_table" type="bool | int" default="false">
  Number of example results to display in a formatted table after evaluation completes. Set to `5` or `10` for a quick sanity check, or `False` to suppress table output entirely.
</ParamField>

---

## Calling the Evaluator

Calling the `evaluate` object with a program runs the full evaluation and returns an `EvaluationResult`. Access `.score` for the aggregate result — a float on a 0–100 scale (e.g., `74.0` means 74%):

```python
result = evaluate(program)
print(f'Score: {result.score:.1f}')  # e.g. Score: 74.0
```

---

## Comparing Programs

The most common pattern is to evaluate your baseline and optimized programs back-to-back using the same `evaluate` instance:

```python
evaluate = dspy.Evaluate(devset=dev, metric=my_metric, num_threads=8)

baseline = evaluate(program)
optimized = evaluate(optimized_program)

print(f'Baseline:  {baseline.score:.1f}')
print(f'Optimized: {optimized.score:.1f}')
print(f'Improvement: +{(optimized.score - baseline.score):.1f}')
```

---

## Inspecting Per-Example Outputs

Every `EvaluationResult` includes a `.results` attribute — a list of `(example, prediction, score)` tuples, one per dataset example. Use it to understand why your program is failing on specific inputs:

```python
result = dspy.Evaluate(
    devset=dev,
    metric=my_metric,
)(program)

for example, prediction, score in result.results:
    if score == 0:
        print('Failed:', example.question)
        print('Predicted:', prediction.answer)
        print('Expected:', example.answer)
```

<Tip>
  Sort `result.results` by score to quickly surface your worst-performing examples and identify patterns in the failures. Targeted failure analysis like this is the most effective way to improve your metric or refine your training data before re-running optimization.
</Tip>

<Note>
  `dspy.Evaluate` is stateless — you can call the same instance on as many programs as you like. There is no need to re-instantiate it between baseline and optimized evaluations.
</Note>
